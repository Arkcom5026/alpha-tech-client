The Delivery Note issue revealed an important operational truth mismatch. — ปัญหาใบส่งของเปิดเผยความไม่สอดคล้องของ operational truth ที่สำคัญ

P1 Document Line / Receipt Decision Update

Discovery:
Receipt can originate from two flows:
1) Cash Sale → Receipt directly
2) Credit Sale → Delivery Note first → Receipt later
Future flow may allow one Receipt to be created from multiple Delivery Notes / Sales.

Decision:
Receipt must be able to add / remove / edit document line text independently from the user’s perspective, because cash receipts may exist without any Delivery Note.

Temporary Patch Rule:
For now, Receipt Short may use existing SaleItem / SaleItemSimple fields:
- documentPrefix
- documentDescription
- documentSuffix

This enables immediate edit support for receipt lines with low backend impact.

Known Limitation:
During temporary patch phase, Delivery Note and Receipt still share the same SaleItem document fields. Therefore editing receipt document text may affect Delivery Note display for the same Sale.

Future Architecture Lock:
When returning for the larger P1 refactor, create document-type-specific truth:
- DeliveryNoteDocumentLine
- ReceiptDocumentLine
- TaxInvoiceDocumentLine
or a generic DocumentLine table with documentType.

Future model must support:
- Receipt from cash sale
- Receipt from one delivery note
- Receipt from multiple delivery notes / sales
- Initial copy/inherit from source document lines
- Independent override after receipt creation
- Traceability back to source SaleItem / SaleItemSimple

ตอนนี้ล็อกแผนไว้แล้วครับ.
