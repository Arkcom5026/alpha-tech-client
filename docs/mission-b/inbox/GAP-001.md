## 1. Current State Machine

```txt
START
  ↓
QuickStock opened
  ↓
Template Catalog search
  ↓
User selects product
  ↓
System classifies selected product
  ├─ Operational Product exists
  │    ↓
  │  OPERATIONAL
  │    ↓
  │  Product detail / BranchPrice editable
  │    ↓
  │  Barcode queue + price validation
  │    ↓
  │  Commit existing-product intake
  │    ↓
  │  POST quick-stock/existing
  │    ↓
  │  STOCK INTAKE COMPLETE
  │
  └─ Template product only
       ↓
     NOT_CREATED
       ↓
     Product detail hidden
       ↓
     User can still enter prices / barcode queue
       ↓
     Commit path is unclear / unsafe
       ↓
     Missing clone/create Operational Product + BranchPrice transition
```

Current state labels observed from reports:

```txt
NO_SELECTION
TEMPLATE_SELECTED / NOT_CREATED
OPERATIONAL_READY
READY_TO_COMMIT_EXISTING_INTAKE
```

The current frontend can distinguish Template from Operational Product, but it does not complete the transition from Template-selected to Operational-ready.

## 3. Missing Transitions

### MT-001 — Template selection to runtime lookup

```txt
Template selected
  ↓
Lookup existing Operational Product for current branch
```

Evidence:
`getOperationalProductByTemplateId()` exists in productApi but is not wired into QuickStockPage according to RT-001.

Impact:
The frontend knows the selected item is Template-only but does not perform the bridge lookup needed to determine operational readiness.

---

### MT-002 — Template-only to create/clone Operational Product

```txt
TEMPLATE_SELECTED_NOT_CREATED
  ↓
Create/Clone Operational Product for current branch
```

Impact:
This is the central missing bridge. Without it, Product Template remains search-only and cannot become store runtime product.

---

### MT-003 — Operational Product creation to BranchPrice creation

```txt
Operational Product created
  ↓
Create BranchPrice for current branch
```

Impact:
The branch product may exist but still not be ready for pricing, checkout, stock intake, or sale.

---

### MT-004 — BranchPrice ready to Operational Ready confirmation

```txt
BranchPrice created
  ↓
UI confirms OPERATIONAL_READY
```

Impact:
User currently cannot verify that the product is truly ready before barcode queue and intake.

---

### MT-005 — Template missing to branch-specific product creation

```txt
No template result
  ↓
Create branch-specific Operational Product
```

Impact:
Mission B cannot be completed until branches can create their own products when no suitable Template exists.

---

### MT-006 — Commit mode separation

```txt
Commit existing intake
```

must be separated from:

```txt
Commit first-time onboarding + BranchPrice + intake
```

Impact:
The same Commit button currently appears to cover different runtime meanings. This creates false completion and unsafe productId risk.

---

## 4. Root Causes

### RC-001 — Missing Product Onboarding State Machine

The current frontend has multiple useful pieces, but no explicit Product Onboarding state machine controlling which transition is allowed.

Current behavior is state-fragmented across:

```txt
selectedProduct
selectedTemplateProduct
operationalProduct
runtimeStatus
runtimeSearchProducts
price fields
barcode queue
```

Without a canonical state machine, Template-selected state can look ready enough for intake.

---

### RC-002 — Runtime and user expectation diverge

RT-001 shows the system does not visibly create Operational Product + BranchPrice from Template.

UI-001 shows the user is told or led to believe that first Commit will clone product, create BranchPrice, and receive stock.

This mismatch is the highest-confidence root cause.

---

### RC-003 — Missing canonical frontend owner for onboarding transition

Potential owners exist:

```txt
QuickStockPage
productStore.saveProduct
productStore.quickStockInAllInOneAction
productApi.getOperationalProductByTemplateId
productApi.quickReceiveExistingProductApi
```

But no single canonical frontend path owns:

```txt
Template → Operational Product → BranchPrice → Operational Ready
```

---

### RC-004 — Branch-created product path is absent from visible UI

UI-001 confirms that when Template search has no result, there is no visible path to create a branch-specific product.

Mission B requires this capability.

---

### RC-005 — Commit readiness is not gated by operational readiness

The user can proceed to price/barcode/commit screens after selecting a Template. Readiness appears tied to selectedProduct, price readiness, and queue readiness rather than a user-visible Operational Ready state.

---

## 5. Frontend-only Recommendations

Implementation remains locked. These are frontend planning recommendations only.

### FR-001 — Introduce explicit Product Onboarding states

Recommended state names:

```txt
NO_SELECTION
TEMPLATE_SELECTED_NOT_CREATED
CHECKING_BRANCH_PRODUCT
OPERATIONAL_READY
BRANCH_PRODUCT_CREATING
BRANCHPRICE_CREATING
INTAKE_READY
INTAKE_COMMITTING
COMPLETE
ERROR
```

Purpose:
Prevent Template-only products from entering intake behavior as if they were operational.

---

### FR-002 — Add explicit user decision for Template-selected products

When Template is selected and Operational Product is missing, the UI should not silently imply readiness.

User-visible decision should be explicit:

```txt
Create product for this branch / Clone to branch
```

Only after success should the UI transition to OPERATIONAL_READY.

---

### FR-003 — Add branch-specific creation path for no-template result

When search returns no result, user should see:

```txt
Create branch-specific product
```

This should create an Operational Product for the current branch and then BranchPrice.

---

### FR-004 — Separate onboarding commit from existing intake commit

The frontend should distinguish:

```txt
Existing Operational Product Intake
```

from:

```txt
First-time Product Onboarding
```

This prevents false completion and incorrect productId usage.

---

### FR-005 — Gate barcode queue and intake by Operational Ready

Barcode queue and Commit should require:

```txt
operationalProduct.id
branchPrice ready or first-create payload ready
```

not merely selectedProduct.

---

### FR-006 — Use existing API surface only after ownership is confirmed

RT-001 identified several existing APIs/actions:

```txt
getOperationalProductByTemplateId
quickStockInAllInOneAction
saveProduct
quickStockIntakeExistingAction
```

Before implementation, ROLE-ARCH must identify which frontend action should be canonical.

## 6. Risks

### R-001 — Template Product ID used as Operational Product ID

If Template-selected product reaches Commit as `productId`, stock intake may target the wrong catalog identity.

Severity: CRITICAL

---

### R-002 — UI overpromises clone-on-commit behavior

ProductMasterPanel tells the user first Commit will clone, create BranchPrice, and receive stock. Runtime evidence does not confirm this behavior.

Severity: HIGH

---

### R-003 — BranchPrice ambiguity

Price fields can mean session intake price, first BranchPrice creation, or existing BranchPrice update. The UI does not clearly separate these meanings by state.

Severity: HIGH

---

### R-004 — Branch-created product missing

Without a Template-missing creation path, branches remain dependent on the Template Catalog and cannot onboard all real-world products.

Severity: CRITICAL

---

### R-005 — Over-expanding implementation scope

The fastest dangerous path would be to touch QuickStockPage, productStore, productApi, ProductFinderPanel, ProductEditor, and backend endpoints all at once.

Severity: HIGH

Control:
One file at a time. Minimal patch. Implementation approval required.

## 8. Architect Verdict

WP-001 and WP-002 together prove that Mission B's core frontend gap is not merely a missing button or missing API call.

The core gap is an absent Product Onboarding State Machine and missing transition ownership between:

```txt
Template-selected
  ↓
Operational Product created
  ↓
BranchPrice created
  ↓
Operational ready
  ↓
Stock intake
```

Mission B should continue with architecture decision work before implementation.

Implementation remains LOCKED.
