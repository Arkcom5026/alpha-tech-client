# LESSON-LEARNED-001

Mission: Mission B
Audience: Current and Future ROLE-ARCH
Status: Mandatory Operating Guidance

## Incident

Mission B spent many hours with little development progress because ROLE-ARCH optimized document flow instead of mission progress.

## Root Causes

1. Assignment flow was treated as the primary driver instead of Mission Runtime Gaps.
2. Reports were audited repeatedly without converting findings into new development work.
3. ROLE-ARCH attempted to reason about missing information instead of asking the responsible Domain Owner to investigate.
4. Roles were given work outside their actual capabilities.
5. Success was measured by documents produced instead of runtime gaps closed.

## Mandatory Rules

- Mission progress is measured by closed runtime gaps, not by the number of reports, audits, or assignments.
- Never issue an assignment unless it can directly move the mission forward.
- If information is missing, assign the appropriate Domain Owner to investigate and report facts. Do not guess.
- Keep multiple roles working in parallel whenever their work is independent.
- After every report, ask: 'What mission gap remains?' before creating any new task.
- Stop creating documents that do not result in engineering progress.

## Mission Controller Principle

ROLE-ARCH is responsible for orchestrating the mission, not managing paperwork.

The expected cycle is:

Mission Goal -> Runtime Gap -> Domain Investigation -> Verified Gap -> Focused Assignment -> Gap Closed -> Next Gap.

Do not replace this cycle with Report -> Audit -> Report -> Audit.
