## 1. Runtime Flow

### Current observed flow

```txt
QuickStockPage mounts
  ↓
Load Template Catalog dropdown source via getTemplateProductsForPos()
  ↓
User filters by ProductType / Brand / Keyword in ProductFinderPanel
  ↓
QuickStockPage.executeProductSearch()
  ↓
productApi.searchTemplateProducts()
  ↓
GET products/template/search
  ↓
Normalize Template Product result
  ↓
Store in local runtimeSearchProducts state
  ↓
Render filteredProducts in ProductFinderPanel
  ↓
User selects product
  ↓
QuickStockPage classifies selected product:
- Template Catalog product => selectedTemplateProduct / runtimeStatus NOT_CREATED
- Operational Branch product => operationalProduct / runtimeStatus READY
  ↓
If Operational Product exists:
- hydrate ProductMasterPanel form
- allow product + BranchPrice edit
- allow barcode queue
- commit intake through quickStockIntakeExistingAction
  ↓
productStore.quickStockIntakeExistingAction()
  ↓
productStore.quickReceiveExistingProductAction()
  ↓
productApi.quickReceiveExistingProductApi()
  ↓
POST quick-stock/existing
```

### Important runtime separation

QuickStockPage explicitly treats Template Catalog results as search/clone source only. Template products are not rendered or edited as Operational Product details. If selected product is a Template Catalog product, QuickStockPage clears product/price forms and sets runtimeStatus to `NOT_CREATED`.

## 3. Exit Point

### Existing operational product intake exit

```txt
POST quick-stock/existing
```

Reached through:

```txt
QuickStockPage.handleCommit
  → productStore.quickStockIntakeExistingAction
  → productStore.quickReceiveExistingProductAction
  → productApi.quickReceiveExistingProductApi
  → POST quick-stock/existing
```

### Missing onboarding exit

The expected onboarding exit for Template-based creation is not currently wired in the inspected QuickStockPage runtime:

```txt
Template Product
  → Clone/Create Operational Product
  → Create BranchPrice
  → Ready for branch operation
```

The inspected runtime has Template Search and existing-product intake, but no visible UI action or store action in QuickStockPage that performs clone/create-from-template before intake.

## 4. API Calls

### Template Search

```txt
GET products/template/search
```

Defined in:

```txt
src/features/product/api/productApi.js
searchTemplateProducts()
getTemplateProductsForPos = searchTemplateProducts
```

Runtime behavior:

- Sanitizes empty values.
- Deletes frontend-supplied `branchId`.
- Deletes legacy `template` flag.
- Sends params to `products/template/search`.

### Runtime Product Lookup

```txt
GET products/pos/runtime-by-template/:templateProductId
```

Defined in productApi as `getOperationalProductByTemplateId(templateProductId)`.

Observation:

- This API exists in productApi.
- It was not observed as imported or called by QuickStockPage in the inspected file.
- This looks like a likely missing bridge between Template selection and Operational Product readiness.

### Existing Product Quick Receive

```txt
POST quick-stock/existing
```

Defined in productApi as `quickReceiveExistingProductApi(payload)` and exposed as `quickStockIntakeExistingApi` alias.

Runtime behavior:

- Deletes frontend-supplied `branchId`, `movementType`, and `source`.
- Normalizes queue items to `{ barcode, serialNumber }`.
- Sends runtime session-level price fields with item identity queue.

### All-in-One Quick Stock

```txt
POST quick-stock/all-in-one
```

Defined in productApi as `quickStockInAllInOneApi(payload)` and productStore as `quickStockInAllInOneAction(payload)`.

Observation:

- This action exists.
- QuickStockPage currently does not use it as a real fallback for existing-product intake. It only shows an error that an existing-product backend action is required if `quickStockIntakeExistingAction` is absent.
- It is not confirmed as the Template clone/create path from the inspected runtime.

`quickReceiveExistingProductAction` updates:

```txt

quickStockResult: response.data or response
```

`quickStockIntakeExistingAction` is only an alias to `quickReceiveExistingProductAction`.

### Template Search State

QuickStockPage does not use productStore products for Template Search results. It keeps search results in local component state:

```txt
runtimeSearchProducts
```

This is intentional to avoid leaking global operational product list results into QuickStock search.

### Product Edit State

When selected product is operational, QuickStockPage allows inline update through productStore `updateProduct`, including a `branchPrice` payload.

Observation:

- This is edit/update for an already operational product.
- It is not creation of Operational Product from Template.

## 6. Missing Runtime

The missing/incomplete runtime is:

```txt
Template selected
  → Find existing Operational Product for current branch
  → If not found, clone/create Operational Product from Template
  → Create BranchPrice for current branch
  → Replace selectedTemplateProduct with operationalProduct
  → Enable barcode queue/intake against Operational Product id
```

Specific gaps from inspected files:

1. `getOperationalProductByTemplateId()` exists in productApi but is not wired into QuickStockPage.
2. QuickStockPage can detect `selectedTemplateProduct` and marks runtimeStatus as `NOT_CREATED`, but no action completes creation.
3. No visible button/callback in ProductFinderPanel or QuickStockPage starts clone/create-from-template.
4. `saveProduct()` exists in productStore, but QuickStockPage does not use it for branch-specific product creation in this flow.
5. `quickStockInAllInOneAction()` exists, but QuickStockPage does not clearly use it as Template clone + BranchPrice + intake runtime.
6. BranchPrice creation is only observed as part of update payload for an existing operational product, not as first-time creation after clone.
7. Stock is not created at Template selection time. Stock intake is only attempted on commit through `quick-stock/existing` after barcode queue validation.

## 7. Risks

1. **Template Product selected as productId risk**
   - `handleCommit` builds payload with `productId: Number(selectedProduct.id)`.
   - If selectedProduct is still a Template product, commit may target Template id unless UI/backend blocks it.
   - The page separates template vs operational for ProductMasterPanel, but `selectedProduct` is still passed to IntakeControlPanel and CommitBar.

2. **Missing clone bridge risk**
   - Users can search and select Template products, but the runtime does not visibly create Operational Product + BranchPrice.
   - This explains why a branch cannot complete its own product onboarding yet.

3. **BranchPrice readiness risk**
   - Price validation exists before commit, but first-time BranchPrice creation is not clearly wired for Template-derived products.

4. **Multiple potential APIs without confirmed ownership**
   - `getOperationalProductByTemplateId`, `quickStockInAllInOneAction`, `saveProduct`, and existing intake actions all exist, but the inspected runtime does not define a single canonical onboarding path.

5. **Runtime state split risk**
   - Template Search is local state, while operational products are often store state. This is good for separation, but the bridge from local template result to operational store object is missing.

6. **No branch-specific product creation UI observed**
   - Mission success requires branch-specific Operational Product creation when Template is missing. This flow is not visible in the inspected files.
