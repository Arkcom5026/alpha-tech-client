## 1. Runtime findings

QuickStockPage is now the active runtime owner for the first product onboarding gate. It already has a local onboarding state set: NO_SELECTION, TEMPLATE_SELECTED_NOT_CREATED, OPERATIONAL_READY, INTAKE_READY, INTAKE_COMMITTING, and ERROR_RECOVERABLE.

Current QuickStockPage can search Template Catalog, classify a selected result as template-only or operational, and block template-only items from barcode intake and existing-product intake. This means the certified state gate patch is present in the inspected file.

The current S2 gap is not search or intake gating. The gap is the transition from TEMPLATE_SELECTED_NOT_CREATED to a confirmed branch Operational Product.

Observed current flow:

```txt
Template Search
  -> select Template result
  -> selectedTemplateProduct
  -> TEMPLATE_SELECTED_NOT_CREATED
  -> ProductMasterPanel receives selectedTemplateProduct
  -> IntakeControlPanel receives null selectedProduct
  -> existing-product intake is blocked
```

Observed operational flow:

```txt
Select operational product
  -> operationalProduct
  -> OPERATIONAL_READY / INTAKE_READY
  -> IntakeControlPanel receives operational product
  -> handleCommit uses operationalProduct.id
  -> quickStockIntakeExistingAction
```

## 2. Existing actions/APIs found

### productApi

Existing frontend API functions relevant to S2:

```txt
createProduct(payload) -> POST products
updateProduct(id, payload) -> PATCH products/:id
addProductPrice(productId, priceData) -> POST products/:productId/prices
quickStockInAllInOneApi(payload) -> POST quick-stock/all-in-one

quickReceiveExistingProductApi(payload) -> POST quick-stock/existing
searchTemplateProducts(filters) -> GET products/template/search
```

`getOperationalProductByTemplateId` exists and is the closest safe lookup API for discovering whether a Template already has a branch Operational Product.

`quickStockInAllInOneApi` is the closest existing all-in-one action by name and endpoint, but its current frontend meaning is not verified as Template-to-Operational creation. It should not be selected as canonical without backend or contract certification.

`createProduct` and productStore `saveProduct` are the closest generic product creation path. However, productStore `saveProduct` deletes `productTemplateId` from payload, so it is not currently a verified Template clone path.

## 3. Missing transition

Missing S2 transition:

```txt
TEMPLATE_SELECTED_NOT_CREATED
  -> check existing branch operational product by template id
  -> if found, replace runtime selection with operational product
  -> if not found, remain template-only and wait for a later approved create step
```

The safest missing first step is lookup, not create.

Reasons:

1. The lookup API already exists in productApi.
2. QuickStockPage already owns the onboarding state machine.
3. Existing state gate already prevents unsafe intake.
4. Product creation and BranchPrice creation ownership are not yet certified.
5. A lookup-first patch can be one file only if it imports the existing API directly.

BranchPrice first creation remains missing. No inspected frontend flow confirms that BranchPrice creation is bundled with Template clone or generic product creation.

## 4. Recommended first S2 patch

Recommended first S2 runtime patch:

```txt
Lookup first: when a Template product is selected, QuickStockPage should call getOperationalProductByTemplateId(templateProductId) and classify the result.
```

Expected behavior for first patch:

```txt
Template selected
  -> CHECKING_OPERATIONAL_PRODUCT
  -> call getOperationalProductByTemplateId(templateProductId)
  -> if operational product exists: set local runtime to operational-ready result
  -> if no operational product exists: keep TEMPLATE_SELECTED_NOT_CREATED
  -> no create/clone yet
  -> no BranchPrice creation yet
```

This patch should not create any product. It should only discover and adopt an already existing Operational Product when backend returns one.

## 7. Risks

### R-001 Template id confusion

If Template id is treated as Operational Product id, stock intake may attach to catalog identity instead of branch runtime product.

Required control:

```txt
Stock intake requires operationalProduct.id only.
```

### R-002 False create assumption

`getOperationalProductByTemplateId` should be treated as lookup only unless backend contract proves otherwise.

### R-003 All-in-one ambiguity

`quickStockInAllInOneAction` may sound like the correct path, but frontend evidence does not prove it creates Operational Product and BranchPrice from Template.

### R-004 BranchPrice readiness ambiguity

Even if Operational Product is found, BranchPrice readiness must be confirmed before sale/runtime readiness in later S2 steps.

### R-005 Scope expansion

Changing QuickStockPage, productStore, productApi, and backend in one round would violate One File at a Time and Minimal Patch policy.
