# UI-001 Report — Product Onboarding User Flow Discovery

Mission: WP-002 User Flow Discovery  
Role: ROLE-RUNTIME  
Implementation Status: LOCKED  
Repository: alpha-tech-client  
Scope: QuickStockPage, ProductFinderPanel, ProductMasterPanel, ProductEditor, IntakeControlPanel, BarcodeScanner, CommitBar

---

## 1. User Journey

### Current user-perceived journey

```txt
User opens QuickStock / Quick Receive screen
  ↓
User sees Step 1: ค้นหาสินค้า
  ↓
User filters by ProductType / Brand / Keyword
  ↓
User sees search results and selects a product row
  ↓
UI hides search result and shows selected product summary
  ↓
User sees one of two product states:
  - Operational Product ของร้าน
  - ยังไม่มี Operational Product ของร้าน / NOT CREATED
  ↓
User continues to Step 2: รับสินค้าเข้า
  ↓
User enters cost / retail price / optional prices
  ↓
User scans barcode and optionally serial number
  ↓
User sees queue summary and Commit button
  ↓
User clicks Commit
```

### What the user likely believes

From the screen wording, the user likely believes this is a single onboarding-and-receiving workflow:

```txt
Search product
  ↓
Select product
  ↓
Enter prices
  ↓
Scan barcode
  ↓
Commit
  ↓
Product becomes ready for this branch
```

This expectation is especially strong because ProductMasterPanel tells the user that when committing the first time, the system will clone the product into the branch, create/save BranchPrice, and receive stock into runtime.

## 2. Screen Flow

### Screen A — ProductFinderPanel

Purpose visible to user:

```txt
1. ค้นหาสินค้า
```

Visible controls:

- ProductType dropdown
- Brand dropdown
- Keyword input
- Search button
- Search result list
- Selected product summary
- Show search result button

User interpretation:

The panel says the user is searching products, but it does not clearly say that the results are Product Templates rather than branch Operational Products. The result rows display name, brand, product type, unit, and mode. They do not display whether the product is Template, already cloned to branch, missing BranchPrice, or already operational.

### Screen B — ProductMasterPanel

There are three user-facing states:

1. No product selected

```txt
เลือกสินค้าก่อนเพื่อแสดงข้อมูล Operational Product ของร้าน
```

2. Template selected, Operational Product missing

```txt
ยังไม่มี Operational Product ของร้าน
NOT CREATED
รายการนี้มาจาก Template Catalog (T01) ใช้สำหรับค้นหาและ Clone เท่านั้น
```

The panel also tells the user:

```txt
เมื่อ Commit ครั้งแรก ระบบจะ:
✓ Clone Product เข้า Branch ปัจจุบัน
✓ สร้าง/บันทึก BranchPrice ของร้าน
✓ รับสินค้าเข้า Stock Runtime
```

3. Operational Product exists

```txt
Operational Product ของร้าน
OPERATIONAL
```

In this state, the user can edit product detail and price through ProductEditor.

### Screen C — ProductEditor

Only reachable when selected product is operational and edit mode is active.

Visible fields:

- Product name
- ProductType
- Brand
- Unit
- Track SN
- Active
- Cost reference
- Retail price
- Technician price
- Online price
- Wholesale price

User interpretation:

This appears to be branch Product Detail / BranchPrice editing for a product that already exists in the branch. It does not support creating a branch-specific product when no Template exists.

### Screen D — IntakeControlPanel / BarcodeScanner

Visible fields:

- Barcode scanner
- Auto-focus serial toggle
- Cost price required
- Retail price required
- Optional wholesale/technician/online prices
- Note

User interpretation:

Because this screen remains active after selecting a Template product, the user likely believes entering prices and scanning barcode are valid next steps even when Operational Product is still not created.

### Screen E — CommitBar

Visible state:

- Number of queue items
- Price readiness
- Queue readiness
- Reset button
- Commit button

User interpretation:

The Commit button appears to be the final action that completes onboarding and receiving. The button does not distinguish between:

```txt
Commit existing Operational Product intake
```

and

```txt
Commit first-time Template clone + BranchPrice + intake
```

## 3. User Decision Points

1. **Which product type should I choose?**
   - User chooses ProductType from Template-derived dropdowns.
   - UI does not explain Template Catalog scope at the search panel level.

2. **Which brand should I choose?**
   - Brand list depends on Template Catalog filtering.
   - User does not see whether the brand/type combination exists in the branch already.

3. **Which product result should I select?**
   - Result rows do not show Template vs Operational status.
   - User only learns NOT CREATED after selecting the product.

4. **What should I do after selecting a Template?**
   - ProductMasterPanel says Commit will clone + create BranchPrice + receive stock.
   - User is guided to continue to prices/barcode/Commit.

5. **What prices should I enter?**
   - Cost and retail price are required before commit.
   - User may believe these prices will become BranchPrice if the product is new.

6. **Should I scan barcode now?**
   - BarcodeScanner is enabled when any selectedProduct exists.
   - For a Template-selected state, this suggests scanning is allowed before Operational Product exists.

7. **Is Commit safe now?**
   - CommitBar readiness depends on selected product, price readiness, and queue readiness.
   - It does not show whether the selected product is operational or still template-only.

8. **What if no Template exists?**
   - User sees “ยังไม่มีผลการค้นหา”.
   - There is no visible branch-created product path from the inspected screens.

## 4. Missing User Flow

### Missing flow A — Template exists but branch product does not exist

Expected user flow:

```txt
Search Template
  ↓
Select Template
  ↓
UI clearly says: product is not yet in this branch
  ↓
User chooses: Create product for this branch / Clone to branch
  ↓
User enters BranchPrice
  ↓
System confirms Operational Product is ready
  ↓
User scans barcode and commits stock intake
```

Current UI gap:

The UI shows NOT CREATED and explains expected commit behavior, but it does not give a distinct user action for “Create Operational Product” or a visible confirmation that Operational Product and BranchPrice were created before stock intake.

### Missing flow B — Template missing entirely

Expected user flow:

```txt
Search Template
  ↓
No result
  ↓
User chooses: Create branch-specific product
  ↓
User fills product detail form
  ↓
User fills BranchPrice
  ↓
Product becomes Operational Product for current branch
  ↓
User scans barcode and commits stock intake
```

Current UI gap:

The inspected screens do not show a branch-specific product creation button or form when no Template exists. ProductEditor is only reachable for an already operational selected product.

### Missing flow C — Operational readiness confirmation

Expected user flow:

```txt
After clone/create
  ↓
UI state changes from NOT CREATED to OPERATIONAL
  ↓
User sees product id / branch readiness / BranchPrice readiness
  ↓
User proceeds to barcode queue and commit
```

Current UI gap:

There is no observed intermediate state that confirms clone/create success before intake. The user may proceed directly from Template selection to Commit.

## 5. User Blockers

1. **User cannot intentionally create branch product from Template before receiving.**
   - The UI promises clone-on-first-commit, but does not expose a clear creation step.

2. **User cannot create a branch-specific product when Template is missing.**
   - Empty search result state only says there are no results.
   - No create button or product form is visible from that state.

3. **User cannot verify BranchPrice readiness before Commit for Template-selected product.**
   - Price fields exist, but there is no explicit BranchPrice created / not created status.

4. **User may be allowed to scan barcode while selected product is only Template.**
   - BarcodeScanner only checks whether selectedProduct exists.
   - From a user perspective, this means Template selection looks sufficient.

5. **User cannot distinguish Commit modes.**
   - The same Commit button appears to cover existing-product intake and first-time onboarding.
   - The user does not receive a separate confirmation for clone/create.

6. **Product Detail editing is unavailable exactly when the user may need creation.**
   - For Template-selected state, ProductMasterPanel intentionally hides ProductEditor.
   - This is correct for Template separation, but it leaves no alternate branch product creation form.

---

## 6. UX Risks

1. **False completion risk**
   - The user may believe that selecting a Template and seeing the selected product summary means the product is ready for the branch.

2. **Commit expectation mismatch**
   - ProductMasterPanel says first Commit will clone, create BranchPrice, and receive stock.
   - If runtime does not actually perform this end-to-end, the UI overpromises.

3. **Template vs Operational ambiguity in search results**
   - ProductFinderPanel does not label result rows as Template Catalog items.
   - The user only sees NOT CREATED after selection.

4. **Branch-created product path is invisible**
   - Mission success requires branch-specific product creation when no Template exists.
   - The current empty state does not guide the user to create one.

5. **Price meaning ambiguity**
   - The same price fields may mean intake session price, BranchPrice first creation, or Operational Product price update depending on state.
   - UI does not clearly separate these meanings.

6. **Operational readiness not visually gated**
   - Barcode queue and Commit readiness use selectedProduct rather than a user-visible operational-ready state.
   - This can mislead the user into treating Template as operational.

7. **Recovery/delete action proximity risk**
   - The operational state includes a recovery delete action near normal edit controls.
   - Although labeled, it may be dangerous in a high-pressure stock recovery screen.
